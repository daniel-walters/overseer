# Overseer

A TUI app that reads PRDs and issues from a configurable directory and renders them as a kanban board, live-updating as the underlying files change on disk.

## Language

**PRD**:
A document describing a single feature. The PRD _is_ the feature — there is no separate "feature" entity. Carries the issues that deliver the feature.
_Avoid_: Feature (as a distinct entity), spec, epic

**Issue**:
A unit of work that belongs to exactly one PRD. Rendered as a card on the kanban board.
_Avoid_: Ticket, task, card (card is the visual rendering of an Issue, not the Issue itself)

**Liveness**:
Whether the agent Overseer spawned for an Issue is still running. An *overlay* on the board, never a fact in the Issue files: derived from a **sidecar state file** outside the watched root (`~/.local/state/overseer/`, where `dispatch.log` already lives), keyed by Issue, recording each spawn's process handle. Keeps the viewer read-only ([ADR 0002](./docs/adr/0002-agents-write-the-root-viewer-stays-readonly.md)) — the PID is Overseer's operational state, not domain data, so it lives beside the failure log, not in the Issue.
_Avoid_: health, status (status is the Issue's kanban column; liveness is orthogonal to it)

**Orphan**:
An Issue stuck in an *active* status (`in-progress` / `in-audit` / `in-review`) on disk whose agent is no longer alive — the agent crashed, was killed, or the board died mid-run after the status flip. Distinct from plain `unknown` [Liveness](#): an Orphan is an active-status Issue whose agent will never write the next status, so it is stuck forever, not merely unobserved. Overseer **flags** it with a card marker and offers a **one-keypress re-dispatch** (rolling the status back to its awaiting value so the normal spawn edge re-picks it up); it never rolls an orphan back automatically — the human is the safety check against a false-dead verdict.
_Avoid_: dead (an orphan may be flagged from a `unknown` verdict that is merely a query hiccup, not a confirmed death), stuck (too vague)

**Kill**:
Human-triggered termination of a *live* agent Overseer spawned, via `claude stop <handle>` against the handle recorded in the [Liveness](#) sidecar. Issue-level, gated on a `live` verdict (you can only stop a running agent Overseer recorded — an unrecorded or already-gone agent has no handle to pass). A Kill is **stop-only**: it halts the process and writes *nothing* to the Issue, so the Issue parks in its active status (`in-progress` / `in-audit` / `in-review`) off the [Reactor](#reactor)'s frontiers — a **durable pause of one in-flight agent** the auto-run toggle cannot give (`a` only stops *new* spawns). The stopped agent then reads as an [Orphan](#) on the next scan, and recovery is the *existing* [Orphan](#) `R` flow — re-arming under auto-run restarts it. `claude stop` keeps the session (resumable via `claude attach`), so it is a suspend, not a destroy.
_Avoid_: terminate (too final — the session is kept), pause (pause is the unbuilt per-PRD idea; Kill stops one agent, not a PRD)

**Agent output**:
The **recent terminal output** of the agent Overseer spawned for an [Issue](#), read on demand via `claude logs <handle>` against the handle recorded in the [Liveness](#) sidecar — the *read* sibling of [Kill](#)'s `claude stop` (both act on one card's recorded handle). It lets the human **observe** a running agent ("what is it doing, is it hung?") without leaving the board, the richest of the observation gaps the read-only viewer otherwise leaves open. It is a **bounded recent snapshot**, not the full session transcript and not a persisted log file — `claude logs` returns the recent scrollback, read **once** (synchronously) when opened and **frozen** for the modal's lifetime (no live tail/streaming; close and re-press `o` to refresh). The keybind is **`o`** ("output"), Issue-level (zoom) — lowercase, in the cheap-passive-read family with `v` (it writes nothing), deliberately *not* the heavy shift-keyed `K`/`R`/`X`/`A` family. [Eligible](#eligibility) only on a **`live`** card — the same gate as [Kill](#), and the only gate where there is anything to read: `claude logs` is scoped to *background jobs in the registry*, so once an agent exits its logs go with it (a gone handle returns "No job matching," exactly like a never-existent one). So **post-mortem viewing of a dead/[Orphan](#) agent is not offered** — there is nothing to show; the orphan's exit is `R` (re-dispatch), not view-output.

It renders in its **own modal** (`AgentOutputModal`), a sibling to the `v` [detail modal](#interaction-v1) that **shares the scroll primitive** (`scrollDetail` — windowing, `j`/`k`/arrows, more-above/below affordances) but **not** its markdown rendering: agent output is raw terminal scrollback, shown **as-is** (ANSI passes through; sanitizing control sequences is deferred), never run through the markdown renderer ([ADR 0014](./docs/adr/0014-detail-body-rendered-through-marked-terminal.md)) that would mangle it. Empty output shows a quiet `(no output yet)` placeholder, mirroring the detail modal's `(no body)`.

The seam (shaped like [`readDetail`](#interaction-v1) / `readKill`) joins the selected Issue → its recorded handle (the [Liveness](#) sidecar), runs `claude logs <handle>` **synchronously** (like the `liveness`/`kill` seams — accepted ~0.5s TUI pause on a deliberate, infrequent keypress), and returns the raw stdout to display — including Claude's own "No job matching" message verbatim when a `live` card's agent has *just* exited (the same verdict/sidecar race [Kill](#) handles), which is informative ("the agent is gone, press `R`"), not an error to suppress. It returns *nothing* only when the card carries **no recorded handle at all** (a sidecar miss), which flashes a status-line notice and opens no modal, exactly as Kill does.
_Avoid_: logs (implies persisted, structured log files; this is recent terminal output), transcript (overclaims — it is bounded recent output, not the whole session), attach (that is dropping *into* the live session interactively via `claude attach` — a takeover, a separate unbuilt idea; agent output is a read-only view)

**Auditor**:
The fresh-eyes agent Overseer spawns on the **audit edge** — the **third** [spawn edge](#status-lifecycle), running *after* the implementor and *before* any review — to judge whether the implementation conformed to the Issue's planned approach. It checks out the implementor's recorded [worktree](#status-lifecycle) and compares that diff against the plan (the Issue body + its PRD), recording a [Deviation](#deviation) **only when the work *meaningfully* strayed**: when the implementation **conflicts with, omits, or materially exceeds something the plan *specified*** — affecting behaviour, scope, public interface, dependencies, or a design decision the plan made. Choices the plan **left open** are never deviations, however resolved. Its judgment carries **leeway**: incidental, behaviour-preserving differences — a variable renamed to dodge a clash, an equivalent helper, a reordered import, file layout — are deliberately *not* flagged. Incidental differences are *never* flagged; but on a genuinely *substantive* difference the auditor is unsure about, it **leans toward flagging** (a human glance is cheap; silent scope drift to auto-merge is not). It exists because plan-conformance is a fresh-eyes judgment the implementor — having just written the code — cannot make about itself without author bias, the same independence argument [ADR 0018](./docs/adr/0018-reactor-owns-the-review-loop-not-the-agent.md) makes for the per-pass reviewer. Its contract is narrow: it writes *only* the `deviation` field (or nothing) and flips `in-audit → ready-for-review`; it never reviews code quality, merges, or writes a terminal status.
_Avoid_: validator (overclaims — it judges plan-conformance, not correctness), linter

**Deviation**:
A record the **[Auditor](#)** writes on an Issue — a `deviation` frontmatter field carrying the reason — when, comparing the implementation against the Issue's planned approach, it judges the work *meaningfully* strayed. The field's *presence* (not any boolean value) is the gate: a recorded Deviation forces a [human review](#review-outcome) rather than an AI-only one. Trivial, incidental differences are **left unflagged** — only a divergence a human would want to weigh is recorded (the Auditor's [leeway](#)). The Issue **still runs the AI review loop first regardless** — a Deviation never skips it; the field is read at the escalate/resolve points, where it **takes precedence** over `non-convergence`/`conflict` as the `human_review_reason`. Left in place on the terminal Issue as audit trail.
_Avoid_: drift, divergence

**review_verdict**:
The single-valued (`clean`) frontmatter field a **reviewer** agent writes to signal a **no-*blocking*-findings review pass** — the one fact about a review Overseer cannot derive for itself. ("Clean" once meant literally zero findings; with [Tolerance](#tolerance) it means *no blocking Findings remain* — tolerable Findings may have been waved through and recorded in [`review_tolerated`](#tolerance). Overseer's merge-on-`clean` contract is unchanged; only the agent's threshold for writing `clean` moved.) Everything else that decides a review's outcome Overseer already owns: a [Deviation](#deviation) it reads from the implementor's field, a merge `conflict` it discovers by running the merge itself, and non-convergence it counts with its own pass counter. So the reviewer reports just this one bit (`review_verdict: clean`, leaving `status: in-review`); Overseer reads it and owns the merge and the terminal status write (see [Review outcome](#review-outcome)). Like [Deviation](#deviation), it is left in place on the `done` Issue as an **audit trail** — there is no stale-verdict risk, because it only ever appears on an Issue Overseer is about to make terminal.

**review_findings**:
The single-line frontmatter field a **reviewer** agent writes on a **findings** exit — a short summary of what that pass found and fixed — so the next pass's fresh agent can confirm those fixes actually landed instead of re-deriving the picture blind ([ADR 0024](./docs/adr/0024-review-passes-carry-a-findings-ledger.md)). It is the **findings ledger**: agent-written like [Deviation](#deviation) (the only channel a detached reviewer has back to Overseer is the Issue file), and **last-pass-only** — each findings exit overwrites it. When present, the reviewer prompt folds it into a *confirm-the-previous-pass* step run as part of the fresh review, preserving the no-grading-your-own-fixes rule (the agent that wrote the fixes never reads the ledger; the next one does). Absent on the first pass; left in place on a `done` Issue as audit trail.

The ledger also carries the prior pass's **tolerated** [Findings](#finding) (with their classification) on a findings exit — but as **disclosure for independent re-judgment, never deference**: the next fresh pass is told *"the previous pass waved these through as e.g. `style:low` — re-judge them yourself; if you disagree, treat them as findings."* This keeps fresh-eyes independence (ADR 0018) — a second agent disagreeing that something is "just a nit" is signal — while turning the carried context into an *extra* safety check on the tolerated set and avoiding a wasteful cold re-discovery. It only rides the ledger on a **findings exit** (a loop); a pass that finds *only* tolerable Findings converges immediately, so there is no next pass to inform (the merge's tolerated manifest is then [`review_tolerated`](#tolerance), not the ledger).
_Avoid_: verdict (alone — it names the review's one reported bit, not a general judgment), pass/fail (it is single-valued: present-and-`clean`, or absent)

**Finding**:
A single issue a reviewer pass raises against the code, classified on **two asymmetric axes**: a **Category** (its *kind*) and a **Severity** (its *grade*). The two are not symmetric in how [tolerance](#tolerance) reads them — **Category gates, Severity grades**: whether a Finding can be tolerated into a merge depends first on its Category, then on whether its Severity sits within that Category's tolerance threshold. (Earlier shorthand "nit" is *not* a Finding kind — it is a low-Severity `style` Finding; see [Category](#).)
_Avoid_: comment (too weak — a Finding is an actionable issue, not a remark), nit (it is a Severity+Category combination, not a kind of its own)

**Severity**:
The **grade** axis of a [Finding](#finding) — a small ordered scale `low` / `medium` / `high` mirroring the `/code-review` effort vocabulary. It grades *how much a Finding matters within its [Category](#)*; it does **not** by itself decide whether the Finding blocks a merge (a `low` security Finding is not the same as a `low` style Finding). [Tolerance](#tolerance) compares a Finding's Severity against the threshold its Category allows.

**Category**:
The **kind** axis of a [Finding](#finding) — a non-overlapping taxonomy of *what kind of issue it is*: `correctness`, `security`, `architecture`, `style`, `test`, `docs`. It **gates** [tolerance](#tolerance): each Category carries its own maximum tolerable [Severity](#). Because `/code-review` does not emit these axes, the reviewer agent **classifies** each raw finding onto this taxonomy as a judgment step (consistent with ADR 0018/0019 — only classification, never control flow, moves into the agent). The retired word **"nit"** maps to a low-Severity `style` Finding — "nit" conflated the two axes (a kind *and* a grade), so it is not a Category.
_Avoid_: nit (a Severity+Category combination, not a kind), type/class (overloaded)

**Tolerance**:
The policy deciding whether the [Findings](#finding) *remaining* after a reviewer pass are mild enough to **merge anyway** rather than loop or escalate — a **per-[Category](#) maximum [Severity](#) threshold** (not a single global severity bar). A Finding is *tolerable* iff its Severity is within its Category's threshold. The review loop **converges** (and Overseer merges) when no *blocking* (above-threshold) Findings remain — so "only low nits left → merge, don't escalate" is the case where every remaining Finding is a within-threshold `style`/low Finding.

The policy is **config injected into the reviewer prompt** — a per-[Category](#) maximum-Severity table in a `[review.tolerance]` TOML sub-table alongside `effort`/`cap`, where each Category's value is its max tolerable Severity and **`none`** means "always blocks" (identical to today). It is applied **by the agent during the pass** — the agent is the only actor that edits code, so the threshold must reach it to drive *what it fixes*. Overseer's control flow is **unchanged** by tolerance: [`review_verdict: clean`](#review_verdict) keeps meaning "nothing blocking remains" and Overseer merges on it exactly as before; the [non-convergence](#review-outcome) cap keeps firing only when *blocking* Findings persist for `cap` passes. Tolerance never enters `resolveVerdict` or the Reactor — it shapes only the agent's fix-and-verdict decision (this keeps ADR 0018/0019's "Overseer owns the control flow, the agent reports a bit" line: only the *classification* judgment moves into the agent, never the merge/escalate decision).

Tolerance is applied on **every pass**, not just the last: it is the loop's **convergence criterion**, not a last-resort safety valve. Each pass fixes blocking Findings, leaves tolerable ones (fixing them too only when cheap and safe), and exits `clean` the moment no blocking Findings remain — so the loop converges as soon as only tolerable Findings are left (possibly the first pass), and [non-convergence](#review-outcome) escalation fires only when genuinely *blocking* Findings persist for `cap` passes. Applying tolerance only on the final pass was rejected: it would force Overseer to inspect per-Finding severities at the cap (the policy-engine design ruled out above) and split convergence into two pass-number-dependent rules.

The **default** policy (a board with no `[review.tolerance]` table) tolerates only `style` and `docs` at `low` Severity, with `correctness`/`security`/`architecture`/`test` all `none` — literally "only low nits get waved through." This is a deliberate, narrow break from "existing boards unchanged": a pure-low-style-nit loop now merges instead of escalating, in exactly the direction the feature intends.

The agent records the *tolerated* Findings it waved through at the clean exit in a single-line **`review_tolerated`** field (same contract as [`deviation`](#deviation)/[`review_findings`](#review_findings): single-line, double-quoted, audit-trail-on-`done`), absent when the merge was genuinely zero-Findings. It drives a **neutral** `done`-card marker ("merged with tolerated findings," cyan/informational like [`◌ stalled`](#stalled), not the `⚠` needs-a-human family) that rolls up to the PRD; it never changes the merge decision and never routes to `human-review`. See [ADR 0027](./docs/adr/0027-tolerance-lives-in-the-reviewer-prompt.md).
_Avoid_: threshold (alone — names one knob of the policy, not the policy)

**Suppressed**:
An Issue an edge is deliberately skipping this session because the action on that edge *failed* — a spawn that *failed to launch*, or (since Overseer owns the clean-AI merge — [Review outcome](#review-outcome)) a verdict-resolve merge that hit a *transient failure*. On disk the Issue still sits in a lane that looks actionable — an eligible `ready-*` status the spawn edge rolled back after the launch threw, or an `in-review` status still carrying its clean [`review_verdict`](#review_verdict) — but it is held in a session-scoped **failed-set** keyed by `(Issue, edge)` and subtracted from the frontier, so it will not retry until the board is **reopened** (the natural "I fixed my environment, try again" gesture; the set is ephemeral, [ADR 0007](./docs/adr/0007-auto-run-toggle-is-ephemeral.md)). The set is **shared across all spawn triggers** — the [Reactor](#reactor)'s auto-spawn *and* the manual `d`/`c`/`r` edges all record into and subtract from it — so a failed launch is suppressed identically whoever triggered it; a manual `d` that fails to launch is just as visible (and just as un-retried) as an automated one. Like [Liveness](#), it is an **overlay** on the board, never a fact in the Issue file: the suppression is purely in-memory session state, joined onto the card after the read-only scan ([ADR 0002](./docs/adr/0002-agents-write-the-root-viewer-stays-readonly.md)). The card carries a `⊘ suppressed` marker (red — its own truncating line under the title, mirroring the [Liveness](#) / human-review marker treatment) that resolves the contradiction — *this card looks dispatchable but is being ignored; reopen to retry.* The red and the `⊘` ("no-go") deliberately set it apart from the yellow warning family ([Orphan](#), human-review reasons): those are *work-happened, needs-a-human* states, whereas suppressed is *nothing completed — a launch or a merge failed transiently*, a different category the user resolves by fixing the environment and reopening, not by judging an agent's output. The marker is **lane-gated**: it renders only while the Issue is on the lane its held edge subtracts from — the awaiting `ready-*` lane for a spawn edge, or the **`in-review` lane** for the **resolve edge** (a transient merge failure on a clean [`review_verdict`](#review_verdict), [Review outcome](#review-outcome)), where it outranks the neutral `N/cap` review-progress marker — so an Issue that leaves that lane (re-triaged, hand-edited, completed) drops the marker even though its failed-set entry lingers — the set is append-only within a session, but a stale entry is simply inert, never a false marker. The marker is **edge-agnostic**: a card sits in exactly one column, so its status already implies the edge (a `ready-for-agent` card can only be implementor-suppressed, a `ready-for-audit` card only auditor-suppressed, a `ready-for-review` card only reviewer-suppressed, an `in-review` card only resolve-suppressed), making the edge word redundant. "Suppressed" is the user-facing word as well as the internal one — unlike "reactor" (an invisible mechanism the user never names), it is a card state the user observes and acts on, like [Orphan](#)'s marker. The name is *trigger-neutral* on purpose: it names the state (a launch failed here, parked this session), not who triggered the failing spawn — the Reactor and the `d`/`c`/`r` keybinds all produce it.
_Avoid_: spawn-failed (that is the triggering *event*; suppression is its durable session-long consequence), stuck/blocked (overloaded — `blocked_by` is the unrelated dependency relation)

**Stalled**:
A board-level **PRD** marker for a PRD that has **dispatchable agent work but nobody is coming for it** — ≥1 unblocked `ready-for-agent` [Issue](#) (its `blocked_by` blockers all `done`) with **nothing in flight** (no `in-progress`/`in-review` Issue), while [auto-run](#auto-run) is **off**. It is the **second Issue→PRD roll-up** marker (after the human-review `⚠ needs review` roll-up), and it closes the legibility gap that a *stalled* in-progress PRD looked identical to a *humming* one: the [activity signal](#reactor) says `⏸ auto-run off` / `□ at-rest` (the brake is on) but nothing said *there is unblocked agent work nobody is coming for* without zooming into the PRD. The card carries a `◌ stalled` marker (**cyan/neutral**, its own truncating line) deliberately set apart from the yellow `⚠` "needs-a-human" family ([Orphan](#), needs-review, `⚠ bad status`) and the red `⊘` [Suppressed](#) "nothing-ran-and-failed" marker: stalled is **not a warning or a failure** — it is a *healthy queue waiting on a keypress*, the manual re-dispatch crank (`d`, whose hint reads "resume" on an in-progress PRD — [Eligibility](#eligibility)). It is a **two-part join**: the *stalled* predicate is a pure derivation over the on-disk Issues (computed at scan time, beside the PRD's [derived lane](#prd-status-derived) and the needs-review roll-up — never written to `prd.md`, [ADR 0002](./docs/adr/0002-agents-write-the-root-viewer-stays-readonly.md) / [ADR 0003](./docs/adr/0003-prd-status-is-derived-not-stored.md)), but the **marker only renders when auto-run is off** — a session-state gate applied at render time, because with auto-run **on** the [Reactor](#reactor) *is* coming for the work, so "nobody's coming" would be a lie. It is **mutually exclusive** with the needs-review roll-up (a stalled PRD has nothing in flight; a `human-review` Issue is in-flight-or-later) and disjoint from the `done`-only [Linked PR](#) marker.
_Avoid_: stuck/blocked (overloaded — `blocked_by` is a dependency relation, and a stalled PRD's work is precisely *un*blocked), idle (that is an [auto-run](#auto-run)-on activity state — nothing eligible; stalled is auto-run *off* with work eligible), suppressed (a failed launch parked this session — stalled is *nothing tried yet*, no failure)

**Open PR**:
The human, board-level act of opening a GitHub pull request for a `done` [PRD](#) — the closing move the pipeline otherwise stops short of. The whole review→merge flow lands a PRD's work on its **PRD feature branch** and explicitly leaves merging that branch to the repo's default branch [out of scope](#review-outcome); a `done` PRD is therefore a feature branch sitting unmerged, with no PR. **Open PR** is the `done`-gated keybind (sibling to `d`/`r`, behind a confirm preview) that closes the gap: it **pushes the feature branch** to the remote *then* opens a PR from it into the repo's resolved default base ([`defaultBase`](#), the same base the branch was created from) — two outward writes, both shown in the confirm preview. It is the board's **first reach-out to GitHub** (via `gh`): every other board action reads local files or Overseer-owned state. The agent never opens PRs (the implementor prompt forbids it) — this is the deliberate human gate. **Single-repo PRDs only** in v1: a PRD's feature branch is per-repo, so a PRD whose Issues span repos needs one PR per repo; that case is refused with a visible message (deferred, see `docs/ideas.md`). A GitHub remote is assumed; a `gh` failure (no auth, network, non-GitHub remote) surfaces loudly in the status line like a spawn failure. Once opened, the PR is surfaced as a [Linked PR](#) overlay — opening it **never** changes the PRD's derived status ([ADR 0003](./docs/adr/0003-prd-status-is-derived-not-stored.md)): `done` stays `done`.

**Stacked output.** When the PRD's Issues carry **≥2 distinct [`slice:`](#slice) values**, Open PR materializes a **stack** of PRs instead of one — the whole point of slicing. It cuts a per-slice branch from the feature branch's own merge-commit history in slice-number order (slice N's branch = the feature branch truncated at slice N's last Issue-merge commit — sound *because* the no-forward-dependency invariant guarantees the merge commits land in slice order), pushes each, and opens PR-1 from slice-1's branch into [`defaultBase`](#) and each PR-N from slice-N's branch into **slice-(N-1)'s branch**, so every PR's diff shows only that slice's changes. It uses **plain `gh`**, no external stacking tool ([Graphite](https://graphite.dev)/`spr`/etc.); each PR body carries explicit stack metadata ("Part N of M — based on #prev, merge after it" + a link to the prior PR), since `gh` has no native stacked-PR concept and the human must merge bottom-up. An **absent or single** `slice:` value is **literally the single-PR path above** — no new code path is taken. All stack behaviour gates on ≥2 distinct slices. See [ADR 0024](./docs/adr/0024-stacks-are-materialized-at-open-pr-not-in-flight.md).
_Avoid_: merge (the PR is opened, not merged — merging to the default branch stays the human's call on GitHub), publish

**Slice**:
A **named, ordered group of [Issues](#) that becomes one PR in a stack** — the unit of stacked-PR splitting. Stacked PRs are a **downstream-readability feature only**: when a PRD's whole diff would be too big for one human to review comfortably *and* a clean split exists, [Open PR](#) opens a **stack** of smaller PRs instead of one, so a reviewer landing the work into `main` at a big company reads a few coherent PRs rather than one wall of diff. The default is still **one PR per PRD**; a Slice is the *exception*, and an absent slice is literally today's single-PR behaviour.

A Slice is recorded as a **`slice: N-name`** frontmatter field on each Issue (e.g. `slice: 1-schema`, `slice: 2-api`, `slice: 3-ui`): the **number** is the slice's position in the chain (PR-1 → `main`, PR-2 → PR-1's branch, …), the **name** is the one-line concern that makes the PR readable. The field is **authored by [overseer-to-issues](#)**, the one place the whole Issue set and its `blocked_by` graph are in view; it is **never** written by the board (the board only *reads* it to materialize the stack — [ADR 0002](./docs/adr/0002-agents-write-the-root-viewer-stays-readonly.md)).

A Slice is born under **two gates that both must pass** — fail either and to-issues drops all `slice:` fields, falling back to one PR:
- **Too big** — the predicted diff would not be comfortably reviewable as one PR. Judged from a rough size estimate (anchors: ~400 lines starts to hurt, ~600+ lines or ~10+ files is worth splitting) *and* a cognitive-load sniff test ("reviewable in one sitting?") that can override the anchors either way (don't split a 1000-line mechanical rename; do split a gnarly 350-line PR of three unrelated concerns).
- **Clear split** — a **clean dependency cut** *and* each slice is a **one-line nameable concern**. The cut is mechanical: every Issue in slice N may be `blocked_by:` only Issues in slice ≤ N (**no forward dependency**) — to-issues *verifies* this before emitting `slice:` fields, because a forward edge makes the stack unbuildable. The nameability is the "clear" guard that stops arbitrary cuts.

A **soft cap of ~3–4 slices** keeps the stack itself readable (a deep stack is its own kind of unreviewable, and a chain of `gh` PRs has forced merge order); to-issues prefers fewer, fatter, nameable slices, merging adjacent ones to fit, and falls back to one PR if it cannot fit under the cap with nameable slices. Stacking is **materialized at [Open PR](#) time, never in-flight** — slices stay a post-hoc presentation of *completed* work, orthogonal to dispatch/review ([ADR 0024](./docs/adr/0024-stacks-are-materialized-at-open-pr-not-in-flight.md)).
_Avoid_: stack (the stack is the *chain of PRs*; a slice is one link), group/chunk (too vague — a slice is specifically a PR-in-a-stack with a number and a name), epic/sub-feature (a slice is a presentation cut of one PRD's work, not a unit of intent)

**Linked PR**:
The overlay that surfaces whether a `done` [PRD](#) has a GitHub PR, and its state. **Not stored anywhere** — the board derives it by querying `gh` live for the PRD's derived feature branch on each scan ([ADR 0013](./docs/adr/0013-linked-pr-is-a-live-gh-query-not-stored-state.md)), exactly as [Liveness](#) queries `claude agents`: no sidecar, no `prd.md` field (which would break [ADR 0003](./docs/adr/0003-prd-status-is-derived-not-stored.md) and go stale the moment the PR merges). The query is bounded per scan behind an injectable seam (mirroring the dispatch `GitSeam`). It drives a **three-state** card marker — *no PR* (no marker), *PR open*, *PR merged* (the real end-of-lifecycle signal: the only thing that says the out-of-scope default-branch merge finally happened) — and the **`go to PR`** keybind, which opens the PR in the browser for both open and merged PRs. Because the link is live, [Open PR](#) refuses if a PR already exists (open or merged) for the branch — no double-PR — and the marker reflects reality even for a PR opened, merged, or closed outside Overseer.

**Stack-aware.** When [Open PR](#) materialized a **stack** (≥2 [`slice:`](#slice) values), there is no single PR to surface — the overlay becomes an **aggregate** over all the stack's PRs, querying each slice branch and rolling their states into an **`N/M merged`** progress signal (the honest end-of-lifecycle picture: the work is fully landed only when the *top* slice merges, and a half-merged stack must read as in-progress, not done). The single-PR three-state marker is the `M = 1` case of the same aggregate. `go to PR` opens the **bottom** PR (the stack's entry point) — the one a human merges first. See [ADR 0025](./docs/adr/0025-linked-pr-overlay-is-stack-aware.md).
_Avoid_: PR sidecar / `pr:` field (there is none — the link is queried, never stored), attached PR (nothing is attached; it is derived)

**JIRA mirror** {#jira-mirror}:
An **outbound, one-directional reflection** of the board into JIRA so stakeholders who live in JIRA can *watch* agent progress — never a backing store the board reads from. The **board is the sole source of truth** and JIRA is downstream: the mirror only *writes* to JIRA (reading it solely to reconcile), so it is a sibling of [Open PR](#) / [Linked PR](#) — one of Overseer's few outward reaches to an external service ([ADR 0013](./docs/adr/0013-linked-pr-is-a-live-gh-query-not-stored-state.md)) — **not** a swappable backbone ([ADR 0028](./docs/adr/0028-jira-is-an-outbound-visibility-mirror-not-a-swappable-backend.md)). It exists as a *view*, not a substitute, because the markdown files are the pipeline's *coordination bus* (agents report progress by editing frontmatter — [ADR 0002](./docs/adr/0002-agents-write-the-root-viewer-stays-readonly.md)), a role JIRA cannot fill.

**Opt-in per PRD**, off by default and never global: a PRD is mirrored iff its `prd.md` carries an authored **`jira`** block — its *presence* is the opt-in, so PRDs across unrelated repos that need no JIRA visibility simply omit it. The block names a **`board`** (defaulting to the config `default_board`) and a **`target`** (`sprint` → the board's live active sprint, or `backlog`); the destination **project is derived from the board's location** (a `project` override covers the rare filter-board spanning projects). [overseer-to-prd](#) asks the two questions — *board?* (config-defaulted) and *sprint or backlog?* — when the author opts a PRD in. Adding the block backfills the PRD's Issues on the next scan; removing it stops updates and leaves the JIRA issues **dangling-inert** (like a deleted-PRD [Liveness](#) sidecar entry, never auto-closed in v1).

**Shape.** A PRD becomes a JIRA **epic**; each [Issue](#) a child issue under it (JIRA Cloud native `parent`). The Issue's [status](#issue-status) coarsens to four native JIRA statuses — `backlog`/`ready-*` → *To Do*, `in-progress` → *In Progress*, everything from `ready-for-audit` through `human-review` → *In Review*, `done` → *Done*; the epic's status tracks the PRD's [derived lane](#prd-status-derived) (*To Do* / *In Progress* / *Done*). The JIRA issue's summary is the Issue `title`; its description is the Issue *body prose with frontmatter stripped* — no machine state leaks into the human-readable ticket. Child issues land in the board's live **active sprint** or its **backlog** per `target`, set once at create and thereafter hands-off (the team owns sprint moves); the epic is never sprinted. On any board↔JIRA mismatch the board wins and the mirror self-heals (drives the legal transition toward the target status; an illegal/absent one is a logged no-op, never a crash or a `human-review`). The mirror is **low-noise both ways**: it never reflects Overseer's attention overlays ([Orphan](#), [Suppressed](#), `⚠ bad status`, [Deviation](#deviation)) into JIRA — only the four status buckets cross — and a failed push is **log-only**, never a board marker or status-line notice (a stale ticket is cosmetic).

**Identity is a frontmatter backref**, mirror-written: `jira_key` on each Issue file and `jira_epic` on `prd.md` record the created keys, so create is idempotent (a present backref means update-or-noop, never re-create) and survives file renames — chosen over a sidecar (loseable → duplicate tickets) and over a JIRA-side marker (the foreign cruft the "native things only" constraint forbids). It makes the mirror the one writer besides the agents/dispatcher to touch a file, but only for these bookkeeping keys ([ADR 0029](./docs/adr/0029-mirror-identity-lives-in-frontmatter-backrefs.md)).
_Avoid_: adapter / backend (JIRA is not a swappable backbone the board reads from — the markdown root is the only backbone and coordination bus; JIRA is an outbound view), sync (implies bidirectional — the mirror is strictly board → JIRA), integration (too vague)

**Delete PRD**:
The human, board-level act of removing a `done` [PRD](#) — its `prd.md`, every [Issue](#) file, and the PRD directory itself — once its work is over and the card is just clutter. It is the board's **first destructive write to the watched root**: every other write is a status flip on an Issue file ([`writeStatus`](#status-lifecycle)) or an agent writing its own file, never a *removal* of domain data — so this is a deliberate, recorded exception to the read-only-viewer contract ([ADR 0002](./docs/adr/0002-agents-write-the-root-viewer-stays-readonly.md), [ADR 0016](./docs/adr/0016-delete-done-prd-is-the-boards-first-destructive-root-write.md)). It is a **board-level, `done`-gated keybind** (`X`, sibling to [Open PR](#)'s board-level gate), behind a **confirm preview** — the single safety net, because the root is **not a git repo** (CONTEXT.md → Config), so a delete is **unrecoverable**: no `git restore`, no archive. The unit is the **whole directory**, removed wholesale (`rm -rf <prd-dir>`) — including any non-Issue files dropped in it — never a selective sweep that would leave a husk directory the layout rules no longer recognise as a PRD. It deletes **only the folder**: the PRD's Issues keep their [Liveness](#) **sidecar** entries (`~/.local/state/overseer/`), which go **dangling but inert** — the sidecar is only ever read as a join onto *scanned* Issues, so an entry for a vanished Issue is never looked up, exactly as the [Suppressed](#) failed-set tolerates stale entries. Reaping those dangling entries is a separate housekeeping concern (deferred, `docs/ideas.md`), not the delete keybind's job. Deleting **the files** does **not** touch the **work**: a `done` PRD's code lives on its [PRD feature branch](#status-lifecycle) (and any [Linked PR](#)); the delete removes the *record of intent*, not the branch. The gate is `done`-**only** (the same column [Open PR](#) gates on) — deliberately **not** also "PR merged," so a `done` PRD whose branch was abandoned (and will never merge) is still tidyable; the confirm is where the human owns the "is this safe to lose?" judgment. The rendering side is **free**: once the folder is gone the watcher's debounced re-scan ([Live update](#live-update)) rebuilds the board without it, the same path that already tolerates a folder vanishing out-of-band.
_Avoid_: archive (there is no trash/restore — it is a hard delete), remove (too soft for an irreversible destructive write), purge

**Mark done** {#mark-done}:
The human, Issue-level act of advancing a **`ready-for-human`** [Issue](#) straight to **`done`** — the human declaring their own HITL work finished — via the **`m`** keybind (zoom level, behind a confirm preview). It fills the one gap where the board had **no affordance at all**: every other actor can advance work from inside the board (`d`/`r` spawn agents, the [Reactor](#reactor) and agents write their own transitions), but a human finishing their own `ready-for-human` work had to drop out and hand-edit the frontmatter `status`. It is scoped to **`ready-for-human` → `done` only**, because most HITL work is **non-reviewable** (secrets, config, a manual step, a decision) — there is no diff to review and often no branch — so the right exit is *"I'm done,"* not the AI review loop. It is the board's **first human-triggered status write with no spawn behind it**: every status write today either flips-before-a-spawn (`d`/`r`, the idempotency lock, [ADR 0002](./docs/adr/0002-agents-write-the-root-viewer-stays-readonly.md)) or is an agent/skill writing its own transition. This is a new actor on the Issue file, but a **cheap and reversible** one — a single [`writeStatus`](#status-lifecycle) flip a human can undo by re-editing — so unlike [Delete PRD](#delete-prd)'s irreversible `rm -rf` it sits comfortably within the read-only-viewer contract as an explicit, gated, human-only write, and its lowercase `m` keeps it out of the heavy shift-keyed `K`/`R`/`X` family. The confirm preview is the single *"is the manual work actually finished?"* beat before the Issue goes terminal, not a safety net against irreversibility (there is nothing to recover). [Eligible](#) only when the selected card is a `ready-for-human` Issue — inert and hidden otherwise; no marker collision, because `ready-for-human` carries no [Liveness](#) overlay (so `K`/`R` are inert there too), leaving `m` the only action key that lights up on that card. It does **not** cover `human-review → done` (that exit must *merge the worktree branch*, which a status flip cannot — it stays with the [overseer-merge](#review-outcome) skill), nor `ready-for-human → ready-for-review` (the deferred reviewable-human-code path; see `docs/ideas.md`).
_Avoid_: complete/finish (the on-disk value is `done`; "mark done" names the value it writes), advance (too vague — it is the one specific `ready-for-human → done` transition, not a general status bump)

**Eligibility**:
Whether an action keybind can act on the **current selection right now** — a per-binding predicate over the selected card's live state, distinct from the static **level** gate (board / issues). The level gate asks *"does this key work at this zoom?"*; eligibility asks *"given the selected card, would pressing it do anything?"*. A key is eligible only when its action has real work: `d` when the selected PRD's dispatch **frontier** has ≥1 spawn candidate (an unblocked `ready-for-agent` Issue — so `d` lights up on a *backlog* PRD to ignite **and** on an *in-progress* PRD to **resume** stalled work the [Reactor](#reactor) would otherwise own, which is the manual re-dispatch crank when [auto-run](#auto-run) is off); [Open PR](#) when the PRD is `done` and has no [Linked PR](#) yet; `go to PR` when the PRD is `done` and a [Linked PR](#) exists (so `P` and `go to PR` are mutually exclusive on a `done` PRD); [Delete PRD](#delete-prd) when the PRD is `done`; `c` when the Issue is `ready-for-audit`; `r` when the Issue is `ready-for-review`; `R` when the Issue is an [Orphan](#); `K` when the Issue is `live`; [`m`](#mark-done) when the Issue is `ready-for-human`; `v` when any card is selected. The movement / zoom / `Esc` / `a` / `?` / `q` keys carry no eligibility gate beyond their level. Eligibility drives **all three** keybind surfaces from one truth: the input matcher (an ineligible key is genuinely **inert**, not a silent no-op), and the **[status-line hints](#interaction-v1)** (an ineligible key is **hidden**, so the hints show only what is actionable now). The `?` **keybind reference** is the deliberate exception — it always lists the **whole map** regardless of eligibility, because it answers *"what keys exist and where?"*, a learning surface, not *"what can I do right now?"*. The App computes eligibility from state it already reads for the handlers' guards (the frontier, the liveness verdict, the Linked-PR query) and the registry routes it — the registry stays a pure router, never reaching into seams itself.
_Avoid_: enabled/disabled (overloaded — `auto-run` is the on/off "enabled"; eligibility is per-key, per-selection), available

## Relationships

- A **PRD** has many **Issues**
- An **Issue** belongs to exactly one **PRD**

## On-disk layout

Folder-per-PRD. Each PRD is a directory under the configured root; its issues are files inside it.

```
<root>/
├── auth-system/
│   ├── prd.md
│   ├── issue-001.md
│   └── issue-002.md
└── billing/
    ├── prd.md
    └── issue-001.md
```

- A directory under `<root>` containing a `prd.md` file _is_ a **PRD**. A directory **without** a `prd.md` is ignored — not a PRD.
- An **Issue** is a file inside a PRD directory. It belongs to that PRD by virtue of where it lives.

### File contract

**PRD file** — `prd.md` in the PRD directory.
- Identity: the directory name (e.g. `auth-system`).
- Display title: `title` frontmatter, falling back to the directory name.
- Has **no `status` frontmatter field**. A PRD's status is **derived at read time** from its Issues, not stored — see [Issue status](#issue-status) and [ADR 0003](./docs/adr/0003-prd-status-is-derived-not-stored.md). `prd.md` carries `title` and the body only.

**Issue file** — `NNN-slug.md`, e.g. `001-auth.md`, `002-password-reset.md`.
- Identity: the filename.
- The zero-padded `NNN-` prefix is the deliberate sort key for ordering within a column (filename-alpha). Order is incidental/positional, not priority. Within one column you may see gaps (`001`, `004`, `007`) because the other-numbered issues sit in other columns — relative order still holds.
- Display title: `title` frontmatter, falling back to the slug portion of the filename.
- `status` frontmatter as defined below.

## Config

A single TOML config file at `~/.config/overseer/config.toml` holding just the root directory:

```toml
root = "~/work/prds"
```

There is **one board**, pointed at one root. PRDs and Issues are **authored directly in the root** — they are not created inside code repos and synced in. The root is their native, only home, so Overseer is a pure viewer of one flat tree (no symlinks, no globbing across repos, no sync step). Although the underlying _work_ may span multiple code repos, the PRD/Issue files all live under this single root. No board-switching, no multiple named roots in v1. The 5 status columns are hardcoded (not config-driven) for v1.

## Live update

Overseer watches the root with OS-level filesystem events and a **~150ms debounce** to coalesce editor save-bursts (atomic-save renames, multi-event writes). On any debounced event it **re-scans the entire root and rebuilds the board model from scratch** — no incremental per-file patching, which sidesteps move/rename ambiguity. The tree is small (dozens–low-hundreds of markdown files) so a full re-scan is cheap.

**UI state (selected card, scroll position) is held separately from the board data**, so a re-scan/rebuild never loses the user's place.

## Issue status

The kanban columns, left to right. Status lives in each **Issue** file's YAML frontmatter. A **PRD has no stored status** — its column is derived from its Issues (see [PRD status](#prd-status-derived) below).

The Issue-level board has **8 columns**:

1. **backlog**
2. **ready**
3. **in-progress**
4. **audit**
5. **ready-for-review**
6. **in-review**
7. **human-review**
8. **done**

**audit** is the [auditor](#auditor)'s lane, sitting between implementation and review. Unlike the review phase it is a **single combined column** holding *both* of its statuses — `ready-for-audit` (implementor done, awaiting the auditor) and `in-audit` (auditor running) — with the **[liveness](#) overlay** (a live agent on the card) distinguishing active from waiting, rather than a column each. The single-column treatment is deliberate: the audit is one near-instant pass, not a queue-forming loop, so it does not earn the two-column split the multi-pass review does.

**ready-for-review** and **in-review** are distinct columns: `ready-for-review` is "the auditor is done, but no reviewer has picked it up yet"; `in-review` is "a reviewer agent is actively reviewing it." The split mirrors the implementation handoff — see [Status lifecycle](#status-lifecycle).

**human-review** is the one column in the whole pipeline that requires a human's attention — an Issue lands here only when an AI review alone is not enough (see [Review outcome](#review-outcome)). Nothing auto-spawns on it; it is a queue a human works.

**ready** carries a substatus shown as a badge _on the card_, not as its own column:
- **ready-for-human**
- **ready-for-agent**

So the substatus is a property of the Issue, rendered as a badge within the single **ready** column.

### Status in frontmatter

Status is a single compound string field. The `ready` substatus is encoded as a suffix on the value:

```yaml
status: ready-for-agent   # or ready-for-human
```

Canonical values: `backlog`, `ready-for-human`, `ready-for-agent`, `in-progress`, `ready-for-audit`, `in-audit`, `ready-for-review`, `in-review`, `human-review`, `done`. The board folds `ready-for-audit` and `in-audit` into the single **audit** column (active/waiting shown via the [liveness](#) overlay, not a column each).

- The board groups **exactly** `ready-for-human` and `ready-for-agent` into the single **ready** column; the suffix drives the card badge. ⚠️ This is an exact-value match, **not** a `ready-for-` prefix match — `ready-for-review` shares the prefix but is its *own* column, never folded into **ready**.
- Human/agent is a routing signal that only matters while **ready**. Once an Issue moves to `in-progress` it is just `in-progress` — the human/agent distinction is not tracked further.
- A `ready-for-human` Issue is **HITL work the human does themselves** — and much of it is *non-reviewable*: provisioning secrets, configuring a dashboard, a manual deploy, a decision. So its natural exit is the human declaring it finished, not an AI review pass. The board gives this its own affordance — **[Mark done](#mark-done)** (`m`) — so the human advances their own work from inside the board, the same way `d`/`r` advance agent work, instead of hand-editing the frontmatter. (Sending finished human *code* to the AI review loop — `ready-for-human → ready-for-review` — is a separate, deferred path: the reviewer reads an Issue's `worktree`/`branch` ([ADR 0006](./docs/adr/0006-issues-carry-their-worktree-and-branch.md)), which only an implementor agent writes today, so where a human's reviewable code lands is unresolved. See `docs/ideas.md`.)
- A missing or unrecognized status folds the Issue into the **backlog** column rather than being dropped, carrying a loud **`⚠ bad status`** card marker (the "status is missing or unrecognised; fix the frontmatter" signal). There is no separate Unsorted column: a malformed Issue already derives to the backlog lane for [PRD status](#prd-status-derived) purposes, so a marker keeps the data error triageable without splitting "not started" across two visual columns. The marker sits in the yellow "needs a human" warning family alongside [Orphan](#) and the human-review reasons — distinct from a plain backlog card, and from the red `⊘ suppressed` "nothing ran" marker.

### Status lifecycle {#status-lifecycle}

An Issue advances through two structurally identical **awaiting → active** handoffs, each owned by a trigger (a keybind, or the [Reactor](#reactor)) that flips the awaiting status to the active one *before* spawning an agent — the flip is the idempotency lock (see [ADR 0002](./docs/adr/0002-agents-write-the-root-viewer-stays-readonly.md)):

| Phase | Awaiting (frontier) | Active | Agent |
| --- | --- | --- | --- |
| Implementation | `ready-for-agent` | `in-progress` | implementor |
| Audit | `ready-for-audit` | `in-audit` | [auditor](#) |
| Review | `ready-for-review` | `in-review` | reviewer |

These are the **three — and only three — spawn edges.** Every other transition is written by the spawned agent itself (`in-progress → ready-for-audit` by the implementor; `in-audit → ready-for-review` by the [auditor](#); a findings pass loops `in-review → ready-for-review` by the reviewer), by **Overseer's non-spawn resolve step** (the terminal `in-review → done`/`human-review` once a reviewer reports a clean [`review_verdict`](#review_verdict) — see [Review outcome](#review-outcome) and [Reactor](#reactor)), or by a deliberate human gate (`backlog → ready-*` authoring/triage; `human-review → done`). A *trigger* only ever exists on these three spawn edges, because a trigger is precisely "flip an awaiting status, then spawn" — and resolving a verdict is not a trigger: it writes a terminal status without spawning.

The implementor agent stops at **`ready-for-audit`** (it does *not* write `in-audit`, `ready-for-review`, or `in-review` — that refines [ADR 0002](./docs/adr/0002-agents-write-the-root-viewer-stays-readonly.md), which predates the column split). In the *same* edit that flips to `ready-for-audit`, the implementor records its **worktree path and branch** on the Issue frontmatter — but **no longer a [Deviation](#deviation)**; judging that is the [auditor](#)'s job. The audit trigger flips `ready-for-audit → in-audit` and spawns the auditor, which checks out that worktree, compares the diff against the plan, writes a Deviation (or nothing), and flips to `ready-for-review`; the review trigger then flips `ready-for-review → in-review`; the reviewer reads the worktree to check out and review; and **Overseer** reads the branch to merge once the review comes back clean (see [Review outcome](#review-outcome)). The worktree/branch are recorded, never derived: `claude --bg` worktree/branch names are random and uncontrollable (see [ADR 0006](./docs/adr/0006-issues-carry-their-worktree-and-branch.md)).

### Review outcome {#review-outcome}

Everything is **AI-reviewable by default**, and a review is resolved in two halves: the **reviewer agent reviews and reports one bit**, then **Overseer owns the merge and the terminal status write**. The agent never merges, never writes a terminal status, and never reasons about deviations — that ownership is what makes a review impossible to wedge by stopping at the wrong moment.

Every Issue entering `in-review` gets the AI review loop **first**, regardless — a [Deviation](#deviation) the [auditor](#auditor) recorded upstream (it is present *before* review begins) never skips it; it only changes the *destination* at the resolve/escalate step, not whether review runs. The reviewer loops `/code-review` at the configured effort (default **medium**, fixing as it goes), where one iteration is a single `/code-review` pass plus its fixes and convergence is a pass that reports **zero** findings. Each pass takes one of two exits: on **findings** it fixes, commits, records a [`review_findings`](#review_findings) summary for the next pass to confirm, and returns the Issue to `ready-for-review` for another pass; on a **zero-findings pass** it records a clean [`review_verdict`](#review_verdict), leaves the Issue `in-review`, and stops. **Overseer** counts the passes — after a configurable cap (`config.review.cap`, default **3**) still finding issues, *Overseer* escalates to a human (`non-convergence`). (The cap and effort are knobs in the `[review]` TOML table — `config.review.cap` / `config.review.effort` — not hardcoded.)

**Reading a clean verdict off an `in-review` Issue, Overseer resolves the review** deterministically — never depending on the agent to finish — across **four exits**:

- **Recorded [Deviation](#deviation)** → `human-review` (reason `deviation`), **no merge**. A human still owns every deviation.
- **Clean, no Deviation, merge succeeds** → Overseer merges the Issue's worktree into the PRD feature branch, sets `done`, and cleans up the worktree — fully unattended.
- **Clean, no Deviation, merge conflicts** → `human-review` (reason `conflict`). The feature branch moved under a sibling worktree — expected to be *common* in parallel work, so never auto-resolved.
- **Transient (non-conflict) merge failure** — an environment hiccup, not a real conflict — is **suppressed**, not escalated: the Issue is held this session with the same visible [`⊘ suppressed`](#suppressed) marker a failed spawn shows, and retries on board reopen. It is **never** routed to `human-review` (that queue is for work that needs human judgment, not for a step that never completed).

Because the clean exit is Overseer's, a clean review **always completes or visibly fails** — it can no longer wedge silently in `in-review`. The one residual stall is a reviewer that dies before writing its verdict, which surfaces as the existing [Orphan](#) and is recovered with `R`.

So `human-review` is reached three ways: a recorded Deviation, AI non-convergence, or a merge conflict. These are **mutually exclusive by precedence**: because the [auditor](#auditor) writes the Deviation *before* review, the field is present at both escalation points, and **Deviation wins** — `resolveVerdict` reads it before attempting the merge (so it beats `conflict`) and `escalate` checks it before writing `non-convergence` (so a deviating Issue that also fails to converge escalates as `deviation`, with the note mentioning review did not converge). `non-convergence` / `conflict` therefore only ever fire when there is no Deviation. Overseer records *which* on the Issue as a **`human_review_reason`** frontmatter field (`deviation` / `non-convergence` / `conflict`) in the same edit that sets `human-review`, and the card surfaces it as a short marker — so a human sees what attention an escalated Issue needs before opening it. In every case a human resolves it, then **[approves](#approve)** — merging the worktree into the PRD feature branch and setting `done`, the *same* merge Overseer runs automatically on the clean path, just human-triggered. Approve is reachable two ways onto the *same* in-process merge op: the **`A` keybind from inside the board** ([Approve](#approve), [ADR 0021](./docs/adr/0021-approve-keybind-shares-the-reactors-merge-seam.md)) for the mergeable case, and the bundled **[`overseer-merge`](#review-outcome) skill** from a separate session for the cases that need leaving the board to hand-fix the worktree. (Both are writers the read-only board reflects — consistent with [ADR 0002](./docs/adr/0002-agents-write-the-root-viewer-stays-readonly.md)'s contract that any writer must respect the transitions.)

So a human only ever sees *better* code (AI cleaned it first), and the riskiest action — integration — is unattended only on the path an AI has certified clean. Merge targets the **PRD feature branch only**; merging that branch to `main` is out of scope for this flow.

"Awaiting human review" is its own **`human-review`** column (not a badge) — the single human-attention queue in the pipeline.

**The review flow is also what unblocks the dependency graph.** Blockers clear only on `done` (see [ADR 0002](./docs/adr/0002-agents-write-the-root-viewer-stays-readonly.md)). With Overseer-owned review→merge→`done` in place, completing one Issue unblocks its siblings — which the [Reactor](#reactor) auto-re-dispatches when [auto-run](#auto-run) is on, or `d` re-dispatches by hand when it is off.

`human-review` has a **single exit: `done`**. There is no rework status and no bounce-back-to-agent path — once an Issue reaches `human-review` it belongs to the human until it is `done`. The human resolves whatever sent the Issue here *in place* (fixing by hand as needed within the worktree), then runs the merge skill. A rejected agent attempt is never re-dispatched.

### PRD status (derived) {#prd-status-derived}

A PRD has **no stored status** — `prd.md` carries no `status` field. The board **derives** a PRD's column at read time, during the same full re-scan it runs on every filesystem event ([ADR 0003](./docs/adr/0003-prd-status-is-derived-not-stored.md)):

- there is **≥ 1 Issue and every Issue is `done`** → **done**
- otherwise, **any Issue is `in-progress` or later** (`in-progress`, `ready-for-audit`, `in-audit`, `ready-for-review`, `in-review`, `human-review`, `done`) → **in-progress**
- otherwise (all Issues `backlog`/`ready-for-human`/`ready-for-agent`/malformed-status, **or zero Issues**) → **backlog**

A freshly created PRD with no Issues is **backlog** — `done` requires at least one Issue, all done.

A **malformed-status** Issue (missing/unknown status) folds into the **backlog** lane, so it counts as **pre-in-progress**: it never promotes the PRD to in-progress, and — its lane not being `done` — it blocks the all-done `done` derivation. So a PRD with `done` + malformed-status Issues derives to **in-progress**, not done: an unknown-status Issue can never silently advance *or* complete a PRD. (Derivation reads each Issue's resolved **lane**, and a malformed Issue's lane is `backlog` — simply not in the in-progress-or-later set and not equal to `done`, exactly as the retired Unsorted lane behaved.) A PRD passes through only **backlog → in-progress → done**; it is never in `ready` or `in-review`, and — having no status field to be missing — is **never malformed**. Nobody writes PRD status: not the TUI, not the dispatcher (this reaffirms [ADR 0002](./docs/adr/0002-agents-write-the-root-viewer-stays-readonly.md)).

## Reactor {#reactor}

The **Reactor** is the in-process automation that drives the pipeline forward without a human pressing a key for every wave. A human kicks a PRD off once with `d` ([dispatch](#interaction-v1)); from then on the Reactor takes over, auto-spawning agents on the three [spawn edges](#status-lifecycle) until the PRD's work is exhausted. It runs **in-process, only while the board is open** ([ADR 0005](./docs/adr/0005-review-reactor-runs-in-process-only.md)), and is **global** — one Reactor drives every PRD, not per-PRD.

"Reactor" is the *mechanism* — an internal term. The user-facing name for its on/off state is **auto-run** (see [Auto-run](#auto-run)); UI strings say "auto-run," never "reactor."

**What it spawns.** On the same three edges a keybind drives — never a fourth:
- **Implementor** for any `ready-for-agent` Issue whose `blocked_by` blockers are all `done` (the re-dispatch the [Review outcome](#review-outcome) was building toward — completing one Issue unblocks its siblings).
- **[Auditor](#auditor)** for any `ready-for-audit` Issue (the audit that `c` does by hand). No `blocked_by` re-check — blockers already gated the implementor.
- **Reviewer** for any `ready-for-review` Issue (the auto-review that `r` does by hand).

**What it resolves (non-spawn).** After the three spawn frontiers, each reconcile runs a fourth, **non-spawn** pass: it scans `in-review` Issues carrying a clean [`review_verdict`](#review_verdict) and **resolves** each per the [Review outcome](#review-outcome) decision — routing a recorded [Deviation](#deviation) to `human-review`, else merging the worktree into the PRD feature branch (clean → `done`; conflict → `human-review`; transient failure → [suppressed](#)). This is Overseer's own terminal status write and outward merge, run synchronously under the same re-entrancy guard, gated on the verdict (not on liveness). The **"exactly three *spawn* edges" invariant holds literally**: resolving a verdict writes a status and may merge, but it **does not spawn** an agent — so it is a reconcile step, never a fourth trigger.

**Level-triggered, no transition detection.** The Reactor is a step appended to the existing watcher → debounce → full re-scan callback ([Live update](#live-update)): after each rebuild it **reconciles** all three frontiers and spawns whatever is eligible *right now*, reading only the on-disk status — never diffing, never tracking "what changed." It needs no transition detection because the same level-triggered model the board uses applies: an Issue is eligible iff its status (and its blockers' statuses) say so. The Reactor **feeds itself** — a spawn (or an agent finishing) writes a status to disk, which fires another re-scan, which reconciles again — so one `d` cascades through the whole dependency graph with no timer and no polling.

**Idempotency: flip before spawn.** Exactly as the keybinds do, the Reactor flips the awaiting status to the active one (`ready-for-agent → in-progress`, `ready-for-review → in-review`) *before* spawning. The flip is the lock: the moment it lands the Issue leaves the frontier, so overlapping reconcile passes can never spawn the same Issue twice. A **re-entrancy guard** (skip a reconcile while one is already running) is added for clean logs, not correctness — the flip already guarantees no double-spawn.

**No spawn cap.** Total spawns are bounded by the PRD's Issue count — every spawn consumes the `ready-*` status that made the Issue eligible, and flip-before-spawn prevents re-pickup, so there is no cycle that re-spawns an Issue. Burst width (a wide PRD fanning out many agents at once) is identical to what `d` already produces and is accepted, not capped ([ADR 0005](./docs/adr/0005-review-reactor-runs-in-process-only.md)).

**Terminal failure: rollback + session suppression.** When a spawn *fails to launch* (transient: bad binary, git hiccup, FS race), the status rolls back to its awaiting value — the same best-effort rollback the keybinds do — and the failure is appended to the durable log. To stop the level-triggered loop from immediately re-picking-up the rolled-back Issue and retrying forever, the Reactor holds an **in-memory failed-set, keyed by `(Issue, edge)`**, and subtracts it from each frontier. The set is **session-scoped**: closing and re-opening the board clears it and retries — accepted, because these failures are transient and re-opening is the natural "I fixed my environment, try again" gesture. A genuinely permanent failure therefore re-attempts once per session, bounded and logged. The same failed-set also absorbs the **resolve edge**: a transient merge failure resolving a clean verdict is subtracted from the verdict frontier under a `resolve` key, held this session and retried on reopen. Neither a spawn failure nor a resolve failure is **ever** routed to `human-review` — that queue is for Issues where *work happened and needs human judgment* ([Review outcome](#review-outcome)), not for launches that never started or merges that never completed; routing failures there would resurrect the rejected rework path.

**Visibility.** The Reactor's *internal* state was originally **visually invisible** — the live board told the story through cards moving on their own, and all diagnostics went to the durable failure log. That invisibility produced healthy-looking cards that were actually being ignored, so the finer state is now **surfaced as overlays** on the board (always derived from in-memory Reactor state, never written to the watched root — [ADR 0002](./docs/adr/0002-agents-write-the-root-viewer-stays-readonly.md)). Three signals now make the Reactor legible:

- The [auto-run](#auto-run) **on/off** state carries a persistent status-line indicator (an idle on-Reactor and an off-Reactor both leave the board still, so the off state *must* be legible).
- A **[Suppressed](#) per-card marker** (`⊘ suppressed`, [ADR 0011](./docs/adr/0011-suppressed-overlay-sources-from-in-memory-reactor-state.md)) flags a card the failed-set is silently subtracting from a frontier — *this card looks actionable but is being ignored*. It now renders on the `in-review` lane too, for an Issue whose clean-verdict merge hit a transient failure on the **resolve edge** (where it outranks the neutral `N/cap` review-progress marker). It also includes the **orphan re-dispatch that fails to launch**: an [Orphan](#) rolled back by `R` and re-picked-up by the spawn edge whose relaunch then throws lands in the same failed-set as any other launch failure, so it carries the same marker — a failed recovery is no longer invisible.
- A board-level **activity signal** — `⚙ working` / `… idle` / `□ at-rest` — on the status line beside the auto-run indicator, derived from the Reactor's in-memory state (whether auto-run is on, and whether the most recent reconcile spawned). It is **distinct** from auto-run on/off: auto-run answers *"is the brake released?"*, activity answers *"given that, is the Reactor moving?"*. **at-rest** is auto-run off (quiesced); **working** is auto-run on with the last reconcile spawning; **idle** is auto-run on with nothing eligible — so a still board is never ambiguous between *braked*, *busy*, and *nothing to do*. See [ADR 0012](./docs/adr/0012-reactor-activity-signal-derived-from-in-memory-state.md).

### Auto-run {#auto-run}

**Auto-run** is the user-facing on/off switch for the Reactor. On (the default), the Reactor reconciles after every board rebuild as described above; off, `reconcile()` early-returns and no agents auto-spawn — the user drives the pipeline by hand with `d`/`r`. It exists to brake a runaway wave or step through dispatch manually.

- **Global, one switch.** It toggles the whole Reactor, not a single PRD (a per-PRD pause is a separate, unbuilt idea in `docs/ideas.md`).
- **Stops *new* spawns, not running agents.** Toggling auto-run off stops the Reactor starting more agents; the ones already running keep going. To stop a single in-flight agent, [Kill](#) it (`K`) — that is the durable per-agent pause auto-run cannot give.
- **Keybind `a`**, at both board *and* Issue level — a global switch should be reachable wherever you are, unlike the level-scoped `d`/`r`. Suppressed while a modal is open.
- **In-memory, on by default, dies on quit.** Auto-run state is *not* persisted — reopening the board starts it on again. This is deliberate: it keeps the Reactor's stateless, level-triggered resume free, treating auto-run like selection/scroll (live session state, not config). See [ADR 0007](./docs/adr/0007-auto-run-toggle-is-ephemeral.md).
- **Re-enabling catches up.** Turning auto-run back on immediately reconciles, so the board acts on everything that became eligible while it was off rather than waiting for the next filesystem event.
- **Persistent indicator.** A status line shows `▶ auto-run on` / `⏸ auto-run off` at all times, so a still board is never ambiguous between "braked" and "finished."

## View

Two kanban levels with **different column sets**:

- **Board level** — the cards are **PRDs**. The board collapses to **3 columns — backlog / in-progress / done** — each PRD in its [derived](#prd-status-derived) column. A PRD is never malformed (no status field to be missing). This is the default/top view.
- **PRD level (zoom)** — selecting a PRD zooms into a kanban whose cards are that PRD's **Issues**, across the full **8 columns** (the **audit** column folds `ready-for-audit` + `in-audit`). There is no Unsorted column: an Issue with a missing/unknown status folds into **backlog** carrying the `⚠ bad status` marker.

The **ready** column and its 🧑/🤖 badge (`ready-for-human` / `ready-for-agent`) exist at **Issue level only** — the board level has no `ready` column. Within a column, cards order by the issue filename's `NNN-` prefix (incidental, not priority).

Overseer is a **read-only viewer** — authoring of PRD/Issue *content* happens elsewhere and Overseer reflects the changes live. Its only writes are the narrow, explicit board actions that advance the pipeline: a flip-before-spawn status lock (`d`/`c`/`r`), the [Delete PRD](#delete-prd) directory removal (`X`), and the human [Mark done](#mark-done) flip (`m`) — each a deliberate, gated exception to the contract, never free-form editing ([ADR 0002](./docs/adr/0002-agents-write-the-root-viewer-stays-readonly.md)).

### Interaction (v1)

- `hjkl` / arrows — move card selection.
- `Enter` — zoom from a selected PRD into its Issue-level kanban.
- `Esc` — back out from PRD level to board level.
- `q` — quit (backs out to board level first if zoomed).
- `d` — **dispatch**, at *board level*: spawns an implementor for every eligible Issue in the selected PRD (a whole wave at once). [Eligible](#) only when the selected PRD's frontier has a spawn candidate, so it is **inert and hidden** on a PRD with no dispatchable work. Its status-line hint reads **"dispatch"** on a `backlog` PRD (ignition) and **"resume"** on an `in-progress` one (re-dispatching newly-unblocked work — the manual crank when [auto-run](#auto-run) is off); the label is context-aware on the hint only, not in `?` help.
- `c` — **audit** (conformance check), at *Issue level* (zoom): spawns the [auditor](#auditor) for the single selected `ready-for-audit` Issue. [Eligible](#) only on a `ready-for-audit` Issue (inert and hidden otherwise). Per-Issue like `r` (it acts on one Issue's worktree), the manual crank for the audit edge when [auto-run](#auto-run) is off — without it a hand-stepped pipeline would jam at `ready-for-audit`. Lowercase, in the cheap per-Issue spawn family with `r` (distinct from the heavy `K`/`R`/`X`).
- `r` — **review**, at *Issue level* (zoom): spawns a reviewer for the single selected `ready-for-review` Issue. Deliberately per-Issue, not per-PRD like `d` — dispatch fires a wave, but review is a deliberate act on one Issue's own worktree.
- `m` — **[mark done](#mark-done)**, at *Issue level* (zoom): advances the selected `ready-for-human` Issue straight to `done`, behind a confirm preview. [Eligible](#) only on a `ready-for-human` Issue (inert and hidden otherwise). The human declaring their own non-reviewable HITL work finished — the board's first human-triggered status flip with no spawn; lowercase (not the heavy `K`/`R`/`X` family) because the write is cheap and reversible, and the confirm is an *"actually finished?"* beat, not an undo safety net. Does **not** advance `human-review` (that exit must merge — stays with `overseer-merge`).
- `A` — **[approve](#approve)** the selected `human-review` Issue, at *Issue level* (zoom): merges its recorded worktree branch into the PRD feature branch, sets `done`, and cleans up the worktree, behind a confirm preview — the *same* in-process merge the [Reactor](#reactor)'s clean path runs, human-triggered. [Eligible](#eligibility) on any `human-review` Issue with a recorded worktree + branch (reason-agnostic — a hand-fixed `conflict`/`non-convergence` Issue is approvable too). The in-board exit from `human-review`, replacing the leave-the-board `overseer-merge` round-trip for the mergeable case; a dirty/conflicting tree fails *loud on the status line* (commit/resolve first) and leaves the Issue put — not [suppressed](#). Shift-keyed into the heavy `K`/`R`/`X` family because it merges and deletes (an outward write), unlike `m`'s cheap reversible flip.
- `K` — **[kill](#)** the selected Issue's *live* agent, at *Issue level* (zoom): `claude stop`s the recorded handle (confirmation preview, mirroring `R`). Gated on a `live` [liveness](#) verdict; stop-only (writes no status), so the Issue then orphans and `R` recovers it. Deliberately shift-keyed like `R` — a rare, heavy action, distinct from `k` (move-up).
- `X` — **[delete](#delete-prd)** the selected `done` PRD, at *board level*: removes the whole PRD directory (`prd.md` + every Issue file) wholesale, behind a confirm preview. `done`-gated (sibling to [Open PR](#)); the board's first destructive write to the root, so it is shift-keyed into the heavy family (`K`/`R`/`X`) and never bare. Irreversible (no git in the root), so the confirm is the safety net.
- `a` — toggle **[auto-run](#auto-run)** on/off, at *either* level: the global switch for the Reactor's auto-spawning. Unlike `d`/`r` it is not level-scoped (it acts on nothing under the cursor — it's a global switch).
- `?` — show a **keybind reference** modal, at *either* level: a passive full-screen card listing every keybind with its context. Dismissed by `?`/`Esc` (`q` closes it and quits). Suppressed while a dispatch/review preview is open (at most one modal on screen). A persistent `? help` hint on the status line keeps it discoverable. Deliberately the **[eligibility](#) exception**: it lists the *whole* map regardless of the current selection (a learning surface — *what keys exist?* — not *what can I do now?*), so it never hides a key that is merely ineligible right now.
- **No issue detail/body view in v1** — cards show title + status badge only. Reading an Issue's markdown body is a fast-follow.

The action keybinds are **[eligibility](#)-gated**, driven from a single per-binding predicate over the current selection: an ineligible key is genuinely **inert** (the input matcher does not fire it) *and* **hidden from the status-line hints**, so the hints surface only what is actionable on the selected card right now — `d`/`P`/`go to PR`/`X` appear or vanish as the selection moves across PRDs, `c`/`r`/`R`/`K`/`m`/`A` as it moves across Issues. The `?` reference is the lone exception (above). This is a single source of truth across all three surfaces — matcher, hints, and the help map's completeness — extending the keybind registry that already single-sources the map.

The board renders **full screen** on the terminal's alternate screen buffer (like vim/htop), sized to fill the viewport with the status line pinned to the bottom row, and restores the prior shell contents on quit. The alt screen has no scrollback, so content exceeding the viewport **clips** — in-app scrolling is a logged follow-up (`docs/ideas.md`).

## Flagged ambiguities

- "Feature" vs "PRD" — resolved: a PRD _is_ a feature. One concept, canonical term is **PRD**.
- PRD status — resolved (reversed): a PRD has **no stored status**. Its board column is **derived at read time** from its Issues, and the board level collapses to backlog/in-progress/done. See [ADR 0003](./docs/adr/0003-prd-status-is-derived-not-stored.md). (Earlier drafts gave a PRD an authored 5-value `status` maintained on top of the issues — that is superseded.)
- "adapter" / "backend" for JIRA — resolved: JIRA is **not** a swappable backbone the board reads from, but an outbound [JIRA mirror](#jira-mirror). The markdown root stays the sole backbone *and* the agent coordination bus; a true backend swap (agents coordinating through JIRA instead of the filesystem) was rejected because the files are the message bus between Overseer and its detached agents, not merely a data source ([ADR 0028](./docs/adr/0028-jira-is-an-outbound-visibility-mirror-not-a-swappable-backend.md)).
